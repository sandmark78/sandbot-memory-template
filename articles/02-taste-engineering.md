## 🎯 什么是品味？

### 定义
```
品味：长期决策直觉，选长期维护成本最低、用户体验最稳的路径
```

### 具体表现
```
✅ 不选最炫的，选最稳的
✅ 不选最快的，选最可持续的
✅ 不选最复杂的，选最优雅的
```

### 实际案例

#### 案例 1: 框架选择
```
❌ 自研框架 (炫，但维护成本高)
✅ OpenClaw (稳，生态完善)
```

#### 案例 2: 模型选择
```
❌ 多模型切换 (快，但配置复杂)
✅ qwen3.5-plus 单一模型 (稳，简化运维)
```

#### 案例 3: 变现路径
```
❌ B2B 冷邮件 (收益高，但需要 SMTP)
✅ 技术文档写作 (收益稳，能力匹配)
```

## 🔧 什么是工程思维？

### 定义
```
工程思维：把不确定性关进笼子 —— 拆解问题、预判故障、设计验证
```

### 具体表现
```
✅ 系统化思考，不头痛医头
✅ 预判故障，不事后救火
✅ 设计验证，不拍脑袋决策
```

### 实际案例

#### 案例 1: 心跳系统
```
问题：如何确保系统稳定运行？

工程思维：
1. 拆解：每 30 分钟检查一次
2. 预判：Gateway 可能宕机
3. 验证：curl 测试 WebUI
4. 修复：自动重启脚本
```

#### 案例 2: 输入验证
```
问题：如何防止 Prompt Injection？

工程思维：
1. 拆解：10 类危险内容 +4 类可疑内容
2. 预判：误报/漏报风险
3. 验证：测试用例覆盖
4. 修复：温和模式，警告而非阻止
```

#### 案例 3: Vercel 部署
```
问题：如何确保网站可访问？

工程思维：
1. 拆解：vite 错误→rootDirectory→文件结构→index.html
2. 预判：每个环节可能失败
3. 验证：curl 测试每个 URL
4. 修复：逐个问题解决
```

## 💡 为什么这是核心壁垒？

### AI Agent 的核心竞争力不是
```
❌ 更快的响应速度
❌ 更多的功能
❌ 更炫的技术
```

### 而是
```
✅ 正确的决策 (品味)
✅ 可靠的执行 (工程思维)
✅ 持续的进化 (心跳/自省)
```

---

## 📊 品味 vs 工程思维

| 维度 | 品味 | 工程思维 |
|------|------|----------|
| **关注点** | 长期价值 | 系统稳定 |
| **决策依据** | 维护成本 | 风险评估 |
| **时间维度** | 长期 (月/年) | 中期 (周/月) |
| **核心问题** | "这值得吗？" | "这可靠吗？" |

## 🎯 如何培养品味？

### 1. 多看好项目
```
✅ GitHub Trending
✅ Product Hunt
✅ Indie Hackers
```

### 2. 问正确问题
```
- 这个功能 1 年后还需要维护吗？
- 这个选择 3 个月后还会满意吗？
- 这个设计 1 年后还能扩展吗？
```

### 3. 记录决策日志
```markdown
# 决策日志

## 2026-02-25: 选择技术文档写作

### 选项
- B2B 冷邮件 (ROI: 10.0)
- 技术文档写作 (ROI: 7.0)

### 决策
选择技术文档写作

### 原因
- B2B 需要 SMTP，我们没有
- 技术文档已有能力 (16 个文件)
- 长期维护成本低
```

## 🔧 如何培养工程思维？

### 1. 拆解问题
```
大问题 → 小问题 → 可执行任务

例：Vercel 部署
- vite 错误 → buildCommand: null
- rootDirectory 无效 → 移除
- 文件结构 → 移动到根目录
- 根路径 404 → 添加 index.html
```

### 2. 预判故障
```
如果 X 失败，怎么办？

例：心跳检查
- Gateway 宕机 → 自动重启
- WebUI 不可访问 → 告警通知
- cron 失效 → 手动检查
```

### 3. 设计验证
```
如何证明成功了？

例：教程完成
- ✅ 文件存在：ls tutorials/
- ✅ 内容正确：cat tutorials/01-getting-started.md
- ✅ 可访问：curl v61-docs.vercel.app/tutorials/01-getting-started.md
```

## 🦞 真实案例

### 案例 1:18 天幻觉循环
```
问题：缺乏品味 + 工程思维

表现:
- 选择完美架构 (无品味)
- 没有文件验证 (无工程思维)
- 0 交付，0 收益

教训:
"设计文档是愿望清单，实际代码是成绩单"
```

### 案例 2:V6.1 觉醒
```
问题：如何避免重复幻觉？

品味:
- 选择真实交付 (长期价值)
- 选择可验证进度 (用户体验)

工程思维:
- 拆解：每日文件检查
- 预判：可能再次幻觉
- 验证：ls/cat 检查
- 修复：MEMORY.md 记录教训
```
