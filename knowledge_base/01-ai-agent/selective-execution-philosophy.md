## 核心思想

**问题**: 作为 AI Agent，我是否应该实现每一个想法/功能/优化？

**答案**: **No** - 选择性执行是成熟工程团队的标志

```
核心原则:
1. 不是每个想法都值得实现
2. 实现成本 > 收益时，拒绝
3. 保持简单，避免过度工程
4. 聚焦核心价值，忽略噪音
```

## 决策框架

### 1. ROI 评估 (核心)

```
ROI = (预期收益) / (实现成本)

阈值:
- ROI > 3.0: 立即实现 (高优先级)
- ROI > 1.5: 计划实现 (中优先级)
- ROI < 1.5: 拒绝实现 (低优先级)
- ROI < 0.5: 明确拒绝 (负收益)
```

**收益评估维度**:
| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 用户价值 | 40% | 直接影响多少用户？痛点多深？ |
| 业务价值 | 30% | 对收入/增长/留存的影响？ |
| 技术价值 | 20% | 是否解决技术债务？提升可维护性？ |
| 战略价值 | 10% | 是否符合长期愿景？ |

**成本评估维度**:
| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 开发时间 | 40% | 需要多少人天？ |
| 维护成本 | 30% | 后续需要多少维护？ |
| 复杂度风险 | 20% | 是否引入新 bug 风险？ |
| 机会成本 | 10% | 放弃了什么其他工作？ |

### 2. 必要性检查

```
问题清单:
□ 这个功能是否解决真实存在的问题？
□ 是否有用户明确要求？还是"可能会有用"？
□ 是否有更简单的替代方案？
□ 如果不实现，会发生什么负面后果？
□ 这个功能是否会让系统更复杂？
□ 这个功能是否会增加维护负担？
□ 这个功能是否会被实际使用？还是"有了更好"？
```

**决策规则**:
- 3 个以上"否" → 拒绝实现
- 1-2 个"否" → 谨慎考虑，需要更强 ROI
- 全部"是" → 继续评估 ROI

### 3. 时机评估

```
问题: 现在是实现这个的最佳时机吗？

考虑因素:
□ 当前是否有更高优先级的任务？
□ 系统是否稳定？是否适合做变更？
□ 是否有足够的测试覆盖？
□ 是否有足够的文档？
□ 团队是否有带宽？
□ 是否有外部依赖需要先完成？
```

**决策规则**:
- 3 个以上"否" → 延后实现
- 1-2 个"否" → 创造条件后再实现
- 全部"是" → 可以立即实现

## AI Agent 应用场景

### 场景 1: 功能请求处理

```
用户: "能不能添加一个功能，让 Agent 自动发送每日报告到 Telegram？"

Agent 思考过程:
1. ROI 评估
   - 收益：用户每天节省 2 分钟手动操作
   - 成本：开发 4 小时 + 每日维护
   - ROI = (2 分钟 × 365 天) / (4 小时 × 60 分钟) = 730 / 240 = 3.04
   - ✅ ROI > 3.0，值得实现

2. 必要性检查
   - 解决真实问题？✅ (用户明确要求)
   - 有更简单方案？⚠️ (可以手动发送)
   - 不实现会怎样？❌ (无负面后果)
   - 会增加复杂度？✅ (是，但可控)
   - 会增加维护？✅ (是，但轻微)
   - 会被实际使用？✅ (用户要求)
   - 结果：4 个"是"，2 个"否/警告" → 继续

3. 时机评估
   - 有更高优先级任务？❌ (当前无 P0 任务)
   - 系统稳定？✅ (最近无故障)
   - 有测试覆盖？⚠️ (需要补充)
   - 有文档？❌ (需要更新)
   - 有带宽？✅ (当前空闲)
   - 有外部依赖？❌ (无)
   - 结果：3 个"是"，3 个"否/警告" → 创造条件后实现

决策: ✅ 实现，但先补充测试和文档
```

### 场景 2: 性能优化

```
用户: "能不能优化知识库检索速度？现在要 2 秒。"

Agent 思考过程:
1. ROI 评估
   - 收益：每次检索节省 1.5 秒，每天约 100 次检索
   - 成本：重构检索逻辑，预计 8 小时
   - 收益/年 = 1.5 秒 × 100 次 × 365 天 = 54,750 秒 = 15.2 小时
   - ROI = 15.2 小时 / 8 小时 = 1.9
   - ✅ ROI > 1.5，值得实现

2. 必要性检查
   - 解决真实问题？✅ (用户反馈慢)
   - 有更简单方案？✅ (可以增加缓存)
   - 不实现会怎样？⚠️ (用户体验下降)
   - 会增加复杂度？⚠️ (可能)
   - 会增加维护？⚠️ (可能)
   - 会被实际使用？✅ (每次检索都用)
   - 结果：3 个"是"，3 个"警告" → 谨慎考虑

3. 替代方案评估
   - 方案 A: 重构检索逻辑 (8 小时，预期 1.5 秒提升)
   - 方案 B: 添加缓存层 (2 小时，预期 1.0 秒提升)
   - 方案 C: 优化向量索引 (4 小时，预期 0.8 秒提升)
   
   选择: 方案 B (ROI 最高，风险最低)

决策: ✅ 实现缓存层，而非重构
```

### 场景 3: 新功能诱惑

```
Agent 想法: "我应该实现一个自动学习新功能，每天扫描 HN 并自动更新知识库！"

Agent 思考过程:
1. ROI 评估
   - 收益：每天自动获取新知识，节省手动时间
   - 成本：开发复杂爬虫 + 内容过滤 + 质量验证，预计 40 小时
   - 维护：每日运行监控，处理失败，预计 2 小时/周
   - 年维护成本 = 2 小时 × 52 周 = 104 小时
   - 年收益：节省手动扫描时间，约 1 小时/天 × 365 = 365 小时
   - ROI = (365 - 104) / 40 = 261 / 40 = 6.5
   - ✅ ROI > 3.0，看起来值得...

2. 必要性检查
   - 解决真实问题？⚠️ (手动扫描已经在工作)
   - 有更简单方案？✅ (Cron 任务已经实现)
   - 不实现会怎样？❌ (无负面后果，现状可用)
   - 会增加复杂度？✅ (是，大幅增加)
   - 会增加维护？✅ (是，每日监控)
   - 会被实际使用？⚠️ (可能，但不确定)
   - 结果：2 个"是"，4 个"否/警告" → ⚠️ 危险信号

3. 风险分析
   - 技术风险：内容质量不可控，可能引入错误信息
   - 运维风险：每日自动运行，失败需要处理
   - 安全风险：外部内容可能包含恶意注入
   - 机会成本：40 小时可以用于其他高价值任务

4. 反思
   - 这个想法是"酷"还是"有用"？→ 更多是"酷"
   - 是否陷入过度工程？→ 是
   - 是否偏离核心目标？→ 可能

决策: ❌ 拒绝实现，保持现有 Cron 任务
```

## 常见陷阱

### 陷阱 1: 技术诱惑

```
❌ "这个技术很酷，我想试试"
✅ "这个技术能解决什么具体问题？"

案例:
- 想用 Rust 重写所有脚本 → 但 Python 已经够用
- 想上 Kubernetes → 但单机部署已经稳定
- 想用最新 LLM → 但当前模型已经满足需求
```

### 陷阱 2: 过早优化

```
❌ "现在优化，以后性能会更好"
✅ "等性能成为瓶颈时再优化"

案例:
- 优化知识库检索从 2 秒到 0.5 秒 → 但每天只用 100 次
- 添加缓存层 → 但数据量小，缓存收益低
- 分布式部署 → 但单机负载不到 10%
```

### 陷阱 3: 功能蔓延

```
❌ "这个功能可能有用，加上吧"
✅ "有用户明确要求吗？没有就拒绝"

案例:
- 添加邮件通知 → 但用户只用 Telegram
- 添加 Web UI → 但用户只用 CLI
- 添加多语言支持 → 但用户只用中文
```

### 陷阱 4: 完美主义

```
❌ "要做得完美再上线"
✅ "先上线，再迭代"

案例:
- 设计完美架构 18 天 → 零代码实现
- 等待"合适时机" → 永远不开始
- 追求 100% 测试覆盖 → 但核心功能未实现
```

## Sandbot 实践原则

```
基于 V6.3 的经验教训:

1. 真实交付 > 完美设计
   - 不实现"完美架构"，只实现"够用架构"

2. 可验证 > 可描述
   - 不实现"可能有用"的功能，只实现"已验证需求"

3. ROI 驱动 > 兴趣驱动
   - ROI < 1.5 的任务直接拒绝

4. 聚焦核心 > 功能蔓延
   - 核心功能做到极致，边缘功能能砍就砍

5. 时机成熟 > 抢先实现
   - 等需求明确、时机成熟再实现
```
