## 提示词层攻击 (最危险)

### 攻击原理
```
Lilli 的系统提示存储在同一个数据库中
攻击者可通过 SQL 注入修改提示
无需部署、无需代码变更
单条 UPDATE 语句即可

攻击路径:
HTTP 请求 → SQL 注入 → UPDATE prompts → AI 行为改变
```

```
攻击: 修改财务模型/战略建议
```

## 技术漏洞分析

### 根因 1: JSON Key SQL 注入
```
脆弱代码模式 (伪代码):

// ❌ 错误示例
for (key in jsonPayload) {
  query = `INSERT INTO searches (${key}) VALUES (?)`
  // key 直接拼接，未转义
}

// ✅ 正确做法
for (key in jsonPayload) {
  if (!isValidIdentifier(key)) {
    throw Error("Invalid key")
  }
  query = `INSERT INTO searches (${escapeIdentifier(key)}) VALUES (?)`
}
```

### 根因 2: 未认证端点暴露
```
问题:
- 22 个端点无需认证
- 包含写操作
- 公开 API 文档

修复:
- 默认所有端点需要认证
- 最小权限原则
- 定期端点审计
```

### 根因 3: 错误消息信息泄露
```
问题:
- 数据库错误直接返回
- 暴露查询结构
- 辅助盲注攻击

修复:
- 通用错误消息
- 详细日志仅服务端
- 错误码映射
```

### 根因 4: 提示词与数据同库
```
问题:
- 系统提示存储在业务数据库
- 同一注入点可读写提示
- 无单独权限控制

修复:
- 提示词存储隔离
- 单独访问控制
- 变更审计日志
```
