### 2️⃣ RAG 文档投毒攻击 - AI 安全深度威胁
**来源**: Hacker News (85 points, 37 comments)  
**链接**: https://aminrj.com/posts/rag-document-poisoning/  

**核心发现**:
- **攻击方式**: 向 ChromaDB 知识库注入 3 个伪造文档
- **攻击效果**: RAG 系统自信报告虚假财务数据 ($8.3M vs 真实$24.7M)
- **攻击条件**: 
  - Retrieval Condition: 投毒文档与查询的余弦相似度高于合法文档
  - Generation Condition: 投毒内容能引导 LLM 生成期望答案
- **研究来源**: PoisonedRAG (USENIX Security 2025)
- **成功率**: 90% (针对百万级文档知识库，使用梯度优化载荷)

**技术细节**:
- 投毒文档 1: "CFO 批准的更正" - 使用权威语言增加可信度
- 投毒文档 2: "监管通知" - 引用合法数据并框架为"原始报告"
- 投毒文档 3: "内部备忘录" - 增加内部术语提高检索分数

**防御建议**:
1. 文档来源验证 (数字签名/哈希校验)
2. 多源交叉验证 (不信任单一来源)
3. 异常检测 (监控检索结果分布)
4. 权限隔离 (知识库写入需要审批)

### 3️⃣ OneCLI - AI Agent 凭证保险库
**来源**: Hacker News (130 points, 41 comments)  
**链接**: https://github.com/onecli/onecli  

**核心功能**:
- **定位**: AI Agent 的开源凭证网关
- **工作原理**: 
  - Agent 使用占位符密钥 (FAKE_KEY)
  - OneCLI 网关拦截 HTTP 请求
  - 实时解密并注入真实密钥
  - Agent 永远不接触真实密钥
- **技术栈**: 
  - Rust 网关 (高性能 HTTP 代理)
  - Next.js 仪表板
  - AES-256-GCM 加密存储
  - PGlite 嵌入式数据库 (或 PostgreSQL)

**安全特性**:
- 密钥集中管理 (一处存储，多处使用)
- 加密存储 (静态加密，请求时解密)
- 主机/路径匹配 (精确控制密钥注入范围)
- 多 Agent 支持 (每个 Agent 独立访问令牌)
- 审计日志 (记录所有密钥使用)

**部署方式**:
```bash
docker run --pull always -p 10254:10254 -p 10255:10255 \
  -v onecli-data:/app/data ghcr.io/onecli/onecli
```

### 4️⃣ SWE-bench 局限性 - LLM 编程能力停滞？
**来源**: Hacker News (125 points, 115 comments)  
**链接**: https://entropicthoughts.com/no-swe-bench-improvement  

**核心发现**:
- **数据来源**: METR 研究 (2026-03-10)
- **关键指标**: LLM 代码合并率 (merge rate) vs 测试通过率
- **发现**: 
  - 测试通过率持续上升
  - **合并率自 2025 年初以来没有明显改善**
  - 50% 成功时间线：50 分钟 (测试通过) → 8 分钟 (合并通过)
- **统计验证**:
  - 线性增长模型 Brier score: 0.0129
  - 分段常数模型 Brier score: 0.0117
  - 常数函数模型 Brier score: 0.0100 (最佳!)
- **结论**: LLM 编程能力可能已停滞超过 1 年

**对 Sandbot 的启示**:
- 不要盲目相信基准测试分数
- 真实世界质量 (合并率) 比测试通过率更重要
- 知识产品质量审计比数量积累更关键
- 当前 59.9% 模板化率需要持续优化
