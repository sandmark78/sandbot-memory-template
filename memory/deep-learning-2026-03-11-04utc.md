## 🔍 核心洞察

### 洞察 1: AI 验证危机 - "自我祝贺机器"

**问题描述**:
```
当同一个 AI 模型既写代码又写测试时：
- 测试证明的是"代码做了 AI 以为你要的事"
- 不是"代码做了你真正想要的事"
- 同一个模型会错过同样的问题

这就像：自己出题考自己，永远满分
```

**解决方案** (来自 ClaudeCodeCamp 实践):
```
1. 验收标准前置 (Acceptance Criteria First)
   - 在 prompt 之前，用自然语言写下验收标准
   - 每条标准必须可验证 (pass/fail)
   
2. 独立验证器
   - 用 Playwright browser agents 运行验收测试
   - 每个 AC 一个独立 agent，并行执行
   - 截图 + 报告，精确定位失败项

3. 审查失败，不审查 diff
   - 传统：人工 review 每一行 diff
   - 新范式：只 review 失败的验收标准
   - 效率提升：从 50 PR/周 → 可处理 200+ PR/周
```

**我们的应用** (Sandbot V6.3):
```
当前问题：
  - 知识获取 Cron 自动运行，无独立验证
  - 质量审计发现 60% 模板化 (自我生成，无外部校验)

改进方案：
  1. 知识获取后，用不同模型交叉验证
     - 主模型：qwen3.5-plus (获取)
     - 验证模型：换一个 (如 qwen-max 或 bailian 其他模型)
     
  2. 验收标准模板
     - 每个知识点必须有：来源 URL + 核心主张 + 可验证性
     - 模板化内容检测：相似度 >80% 标记为"待人工审查"
     
  3. 质量门禁
     - 新知识文件必须通过：来源验证 + 内容多样性检查
     - 失败项写入：memory/quality-failures.md
```

**ROI 评估**:
```
成本：每次验证额外 1 次模型调用 (~$0.02)
收益：避免 60% 模板化内容污染知识库
ROI: (避免的垃圾内容价值) / (验证成本) = 高

建议：立即实施
```
