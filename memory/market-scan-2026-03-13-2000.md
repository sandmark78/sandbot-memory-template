**基本信息**:
```
创始人：Lewis & Edgar (4 年 RAG 经验)

链接：https://runcaptain.com
Demo: https://pg.runcaptain.com (搜索 PG 文章)
```

**核心技术栈**:
```
文本提取:
  - Gemini 3 Pro (图像)
  - Reducto (复杂文档)
  - Extend (基础 OCR)

嵌入模型:
  - 初期：gemini-embedding-001
  - 当前：voyage-context-3 (Contextualized Embeddings)
  - 理由：chunk 编码包含周围文档上下文感知

重排序:
  - voyage rerank-2.5
  - 50 初始 chunks → 15 最终结果 (可配置)

检索策略:
  - 混合检索：Dense embeddings + Full-text search
  - RRF (Reciprocal Rank Fusion) 融合

API 设计:
  - 单端点：/query
  - 支持：URL、S3/GCS、Google Drive、本地目录
  - 访问控制：Metadata filters
  - 引用：自动页码引用
```

**差异化优势**:
```
✅ 零配置 (3 分钟搭建 RAG pipeline)
✅ 持续更新 (自动跟踪最新检索技术)
✅ 生产级 (4 年 scaling 经验)
✅ 学术背景 (Purdue NLP Lab chunking 技术)
✅ 混合检索 (Dense + Full-text + RRF)
✅ 上下文感知嵌入 (voyage-context-3)
```

### 2. Spine Swarm (YC S23) – AI Agents on Visual Canvas

**基本信息**:
```
创始人：Ashwin & Akshay (13 年朋友，NTU ML 背景)
YC 批次：S23 (2023 夏季，比 Captain 早 3 期)
产品定位：多 Agent 可视化协作平台
链接：https://www.getspine.ai
演示视频：https://youtu.be/R_2-ggpZz0Q
```

**核心理念**:
```
"Chat is the wrong interface for complex AI work"

问题：
  - Chat 是线性线程，真实项目不是线性的
  - 无法看到 AI 如何连接各个部分
  - 无法单独纠正某一步而不重跑全部
  - 无法并行探索两个策略

解决方案：
  - 无限可视化 canvas
  - Block 抽象 (LLM、图像、爬虫、表格、幻灯片等)
  - Block 之间可连接，保证上下文传递
  - 模型无关 (单 workflow 可用 OpenAI → Nano Banana → Claude)
```

**多 Agent 架构**:
```
Orchestrator:
  - 任务分解为子任务
  - 分配给专业 Persona Agents

Agent 特性:
  - 专用 Block 类型 (LLM、图像、爬虫等)
  - 可覆盖默认设置 (模型、prompt)
  - 并行执行 (无依赖子任务)
  - 自动上下文传递 (上游 → 下游)
  - 人工介入 (可暂停请求澄清)

Canvas 优势:
  - 持久化、结构化项目表示
  - Agent 存储中间结果到 Blocks (非上下文窗口)
  - 显式结构化交接 (高效消费)
  - 完全可审计 (追踪每个 Agent 决策路径)
```

**基准测试结果**:
```
Google DeepMind DeepSearchQA:
  - 900 问题，17 领域
  - 因果链结构 (每步依赖前一步)
  - Spine Swarm: 87.6% (零人工干预)
  - 方法：移除无关 Block 类型，禁用人工澄清

GAIA Level 3:
  - 发现基准答案错误/模糊 (可审计性优势)
  - 详细方法论文档：https://blog.getspine.ai/
```

**对 Sandbot 的启示**:
```
验证:
  ✅ Sandbot 7 子 Agent 架构方向正确
  ✅ 多 Agent 协作是趋势 (YC 背书)
  ✅ 可视化 canvas 是差异化点

差距:
  - Sandbot: 无可视化 canvas (纯文本/文件)
  - Sandbot: Agent 交接靠文件 (非结构化)
  - Sandbot: 审计能力弱 (无决策路径追踪)

机会:
  - 集成可视化 canvas (参考 Spine 架构)
  - 加强 Agent 交接结构化 (JSON schema)
  - 增加决策路径审计日志
  - 差异化：垂直场景深度 (50 赛道知识)

行动项:
  - [ ] 研究 Spine Block 抽象设计
  - [ ] 设计 Sandbot Agent 交接 schema
  - [ ] 评估 canvas 集成优先级 (P1/P2)
  - [ ] 加强审计日志 (memory/ 目录结构化)
```

## 🧠 用户需求洞察

### 从"Can I run AI locally?"讨论提取 (488 点，131 评论)

#### 痛点 1: 本地模型工作流复杂
```
用户原话:
"I have spent perhaps 100 hours in the last year configuring local models
for coding using Emacs, Claude Code (configured for local), etc.
However, I am retired and this time was a lot of fun for me.
I don't recommend it for others."

洞察:
  - 本地模型配置耗时 (100 小时/年)
  - 仅适合爱好者/退休人员
  - 普通用户需要"开箱即用"方案

Sandbot 机会:
  - 知识库预配置 (零配置 RAG)
  - 场景化模板 (50 赛道即插即用)
  - 文档化最佳实践 (减少用户试错)
```

#### 痛点 2: 内存带宽瓶颈
```
用户原话:
"128 GiB of memory feel incredibly tight.
I need to leave more memory free for system memory than I was expecting.
~80 GiB quantized seems like a better max limit."
"Memory bandwidth limitations for running the models is real!
10B active parameters at 4-6 bit quants feels usable but slow."

洞察:
  - 大内存仍不够用 (128GB → 实际可用 80GB)
  - 内存带宽是瓶颈 (非容量)
  - MoE 模型是解决方案 (激活参数少)

Sandbot 机会:
  - 云端部署 (规避本地硬件限制)
  - MoE 模型优先 (qwen3.5-MoE 变体)
  - 上下文压缩 (Context Gateway 集成)
```

#### 痛点 3: 小模型场景化价值
```
用户原话:
"Small models like the new qwen3.5:9b can be fantastic for local tool use,
information extraction, and many other embedded applications."
"If you want a general knowledge model for answering questions or a coding agent,
nothing you can run on your MacBook will come close to the frontier models."

洞察:
  - 小模型适合专用任务 (信息提取、工具调用)
  - 大模型适合通用知识/编码
  - 混合策略最优 (小模型专用 + 大模型通用)

Sandbot 机会:
  - 7 子 Agent 按任务分配模型
  - 简单任务用小模型 (降低成本)
  - 复杂任务用大模型 (保证质量)
  - 模型路由策略优化
```
